1. **Split multi-lane tasks at the schema level** when injected. Operator (or whoever drafts TASKS.md) writes:

   - Pro: matches the queue's one-task-one-lane model exactly.
   - Con: one cross-lane initiative is now visually two rows in TASKS.md.

2. **Extend the queue applier** with `per_lane_completion` tracking — task carries a set of unclaimed lanes; each `task/done` POST clears that lane's bit; task is fully done only when the set is empty.
   - Pro: TASKS.md stays compact; intent of "this is one initiative" is preserved.
   - Con: applier complexity grows; rendering/dashboard logic needs update; PHASE-VERIFY cross-checks become ambiguous.

Recommend (1). The cost of redundancy in TASKS.md is small; the cost of changing the applier's contract mid-mission is large.
